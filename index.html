Round 02

告訴我最主要的前幾個功能？

Top 5 核心功能

1 · AI Copilot 對話助手

自然語言對話介面，工程師可以直接用對話方式完成複雜的製程分析。背後是 LangGraph 6 階段 orchestrator，會自動選擇工具（查詢、建圖、分析等），支援 SSE 串流即時回應，並有 6 個月的記憶向量搜尋（pgvector）。

2 · 設備監控儀表板 + 根因分析

全廠（Fab）和單機設備的即時監控，支援兩種模式：

  • Fab Briefing：OOC 熱力圖 + AI 自動摘要
  • Tool Deep Dive：下鑽到 SPC / APC / FDC / EC 等 6 個分析標籤

5 分鐘自動刷新，顏色標示設備狀態（Busy/Idle/Down 等）。

3 · 自動巡檢 + 智能告警中心

排程或事件觸發的自動巡檢，條件符合時自動生成告警。提供完整漏斗視覺化（事件 → 技能執行 → 告警），並追蹤每個告警被跳過的原因（重複、測試、未通過等）。

4 · Pipeline Builder + Skill 管理

拖拉式視覺化流程建構，透過組合「Block」（原子操作單位）來建立複雜分析流程。內建 Planner / Builder / Repair 三個 AI Agent 協助自動生成 pipeline，支援乾跑（Dry-run）驗證。

5 · Agent 可觀測性 + 記憶系統

完整的 AI 決策軌跡視覺化，可查看每個 Agent 的思考過程、記憶引用、LLM Token 消耗。記憶系統分 6 大類（領域知識、偏好、修正記錄等），持續學習累積。

一句話總結：這是一個專為半導體/製造業設計的 AI 智能運維平台，讓工程師用對話操控自動化分析流程，並對 AI 決策過程完全透明可查。

Round 06

這會不會有點奇怪，關於你說呼叫哪些 MCP 是單純由一個 model 決定？是有設定特定的 prompt 或 system prompt 嗎？還是直接把 user prompt 丟進去？還是有搭配什麼參數？

另外比較好的做法會不會是一個 MCP 裡面決定呼叫什麼工具比較好？現在有 12 個 MCP，感覺像是 12 個工具的切法？

實際運作方式

不是直接丟 user prompt。有完整的 prompt 結構：

System Prompt = [Block 1: 穩定，可快取] <soul> 人設 + 硬規則 + 路由規則 </soul> + [Block 2: 每次動態注入] <user_preference> 使用者偏好 </user_preference> <skill_catalog> 現有 skill 清單 </skill_catalog> <mcp_catalog> 12 個 MCP 的名稱 + 說明 </mcp_catalog> Tools（Anthropic tool_use API）= 高階工具清單 plan_pipeline / build_pipeline / modify_current_chart / get_current_status / search_skills / activate_skill / ...
關鍵：12 個 System MCP 不是直接當成 tool 傳給 LLM，而是放在 <mcp_catalog> 區塊讓模型「看到」。模型如果要用，是呼叫通用工具 execute_mcp(mcp_name="get_spc_data", params={...})，由 dispatcher 實際打 API。

你的設計問題分析

現在的做法（model 選 MCP）

用戶說「EQP-03 SPC 狀況」
  → LLM 看 mcp_catalog → 決定呼叫 execute_mcp("get_spc_data", ...)
  → dispatcher 打 API

已知問題：LLM 要「記住」12 個 MCP 的名稱和參數，容易選錯或填錯參數。早期版本有 bug：LLM 猜錯欄位名稱（toolID vs equipment_id），燒了好幾輪才修正，程式碼裡的 _mcp_json_schema 就是在解決這個問題。

你說的「讓 MCP 內部決定」— 其實外部介面就是這樣做的

mcp_pipeline_builder/server.py 就是一個「聰明的 MCP」，外部 Claude 只呼叫 build_pipeline(instruction) 這一個高階工具，裡面的 Planner/Builder 自己決定用哪些 block。

這也是為什麼 chat_agent_loop.py 的 _TOOLS 設計成高階工具而非直接暴露 12 個 MCP：

# LLM 看到的是高階意圖工具
plan_pipeline(instruction)       # ← 不是 get_spc_data / get_lot_trace
modify_current_chart(instruction)
get_current_status()

三層設計評估

層級誰決定評估
Chat Agent LLM 選高階工具（plan/modify/status） 合理 工具定義清晰，有 system prompt 路由規則約束
高階工具內部 程式碼或 Planner Agent 決定用哪些 MCP/Block 合理 封裝好，外部看不到細節
execute_mcp LLM 直接選 MCP 名稱 脆弱 有遇到過問題，理想上應再包一層語意路由
結論：你的觀察沒錯——理想上 12 個 MCP 應該再包一層語意路由，而不是讓 LLM 直接查名稱表來決定。這個專案目前是兩者混用：外部走高階包裝，內部部分還是讓 LLM 直接選 MCP name。

1 · Agent 是否可以自主執行多步驟任務

是，而且是核心設計。

  • Chat Agent（chat_agent_loop.py）：Anthropic tool-use loop，最多 8 輪工具呼叫不需要 human in the loop。例如工程師說「分析 EQP-03 最近 SPC 趨勢」→ Agent 自動決定呼叫哪些工具、以什麼順序，最後組裝報告。
  • Builder Agent（graph_build/graph.py）：LangGraph StateGraph，START → goal_plan → agentic_phase_loop → validate → dispatch_op → finalize → END，可自動修復（repair_plan / repair_op 各最多 2 次）。
  • Auto-Patrol（Skill role=patrol）：排程或事件驅動，完全無人觸發，條件符合時自動建 Alarm。

2 · RAG / SQL / Code Sandbox

技術是否使用說明
RAG 是（輕量） pgvector + bge-m3/Cohere 1024 維向量。agent_knowledge 表儲存過去經驗與規則，按語意搜尋注入 context。非主力查詢路徑，屬於輔助補充。
SQL 查詢 不直接用 用 Block DAG（pandas 操作）取代 SQL。製程資料來自 System MCP（HTTP API），不是讓 LLM 生成 SQL。避免 LLM 產生危險 SQL 的風險。
Code Sandbox Mock，尚未上線 /internal/sandbox/run 路由存在但只回 mock response（"Phase 4 mock — real sandbox wires in Phase 5"）。

3 · AWS VPC 部署

從 code 看不到明確的 AWS VPC 設定。目前是 EC2 單主機 + systemd 管理，nginx 做反向代理 + TLS 終止。沒有 ALB、VPC Security Group 等 AWS-specific 設定。docs/deploy-kubernetes.md 有 K8s 遷移計劃但尚未執行。

4 · Multi-model Failover

目前沒有自動 failover。LLM client 是 singleton（get_llm_client() 快取），若 Anthropic API 不可達，所有 Agent 功能都會 fail。但有以下層次的靈活性：

  • LLM_PROVIDER env 可切換 anthropic / ollama / internal-proxy（需重啟）
  • 文件（docs/LLM_BAKEOFF_4MODEL.html）顯示曾測試 Kimi K2、GLM、Qwen3、GPT 等多個模型
  • 不同任務用不同模型（Sonnet 做主 Agent、Haiku 做衍生草稿）但沒有 automatic failover 機制

Round 09

Conventional ML/DL 是什麼？

Conventional ML / DL 的定義

Conventional ML（傳統機器學習）

工程師從資料中提取特徵，再用統計模型學習規則。代表技術：

  • Decision Tree（決策樹）、Random Forest（隨機森林）
  • SVM（支持向量機）、Logistic Regression（邏輯迴歸）
  • K-Means 聚類、PCA 降維
  • 工具：scikit-learn（Python 的主流 ML 套件）

Deep Learning（深度學習）

多層神經網路，從大量原始資料自動提取特徵。代表技術：

  • CNN（卷積神經網路）— 圖像辨識
  • RNN / LSTM / Transformer — 時序、自然語言
  • 工具：PyTorch（Facebook/Meta）、TensorFlow/Keras（Google）

跟這個專案的關係

這個平台是 GenAI（LLM）驅動，不是 Conventional ML/DL。
分析邏輯靠 Block DAG（pandas 規則）+ Claude 推理，不訓練任何預測模型。 LLM 本身是別人訓練好的大型 Transformer，不算平台自己的 DL 工作。
比較Conventional ML/DL這個平台（GenAI）
做什麼訓練預測模型用 LLM 推理 + 規則執行
資料量需求需要大量標記訓練資料主要靠提示詞工程
結果形式模型權重（.pkl, .pt）文字、圖表、結構化報告
典型代表scikit-learn, PyTorchAnthropic Claude API

這個 RAG 的目的不是搜大型知識庫——它是把「過去工程師做過的好案例、提出的規則」存起來，下次遇到語意相近的問題時注入 context，讓 Agent 不用每次重新摸索。資料量小、語意明確，pgvector 的輕量方案完全夠用。

為什麼需要向量搜尋呢？

為什麼傳統 SQL 搜尋不夠用

假設工程師過去存了一條知識：「EQP-08 的 Yield 下降通常與 Step 3 的 DC 值 drift 有關，建議先看 APC 參數。」

今天工程師問：

"偵測 OOC 趨勢，找出可能的設備異常原因"
搜尋方式能找到嗎？原因
SQL LIKE '%OOC%' 找不到 存的是「Yield 下降」「DC 值 drift」，沒有「OOC」關鍵字
全文搜尋（Elasticsearch） 找不到 「偵測」「趨勢」「異常原因」與「Yield 下降」「drift」字面不同
向量搜尋（pgvector） 找到了 兩段文字的語意相近（都在說製程異常偵測）→ cosine similarity 高

向量搜尋的原理（一句話）

把文字轉成 1024 維的數值向量，語意相似的文字在高維空間裡距離近，用 cosine distance 找最近的幾筆，不需要字面完全匹配。

關鍵場景：半導體製程的問題描述方式多樣（中文、英文、縮寫、技術術語交替），向量搜尋讓過去的知識能被準確召回，即使用詞完全不同。

解釋一下 Pipeline 由 Block 組成（DAG 結構）—— 要用實際的專案 code 作為解釋基礎。

Pipeline = DAG of Blocks

什麼是 DAG？

Directed Acyclic Graph（有向無環圖）：每個節點是一個 Block，節點間有方向性的邊（資料從上游流向下游），不允許循環。

一個實際的 Pipeline 例子

block_process_history  →  block_filter  →  block_step_check  →  block_line_chart
（查製程歷史）           （篩選條件）       （判斷是否超標）       （畫折線圖）

executor.py — 自製 DAG 執行引擎（核心）

不用 Airflow、Celery 任何框架，純 Python collections.deque 做拓撲排序：

# python_ai_sidecar/pipeline_builder/executor.py（節錄）
from collections import deque

# 建 in-degree 表
in_degree = {n: 0 for n in nodes}
for edge in edges:
    in_degree[edge.target] += 1

# 拓撲排序 queue：從 in_degree==0 的節點開始
queue = deque([n for n in nodes if in_degree[n] == 0])

while queue:
    node_id = queue.popleft()
    executor = registry.get(node_id)        # 取得對應的 BlockExecutor
    result = await executor.execute(
        params=node.params,
        inputs=upstream_outputs[node_id],   # 上游的輸出（通常是 DataFrame）
        context=ctx
    )
    outputs[node_id] = result               # 存起來給下游用
    for downstream in graph.successors(node_id):
        in_degree[downstream] -= 1
        if in_degree[downstream] == 0:
            queue.append(downstream)

Block 間的資料傳遞

每個 Block 的 execute() 接收 inputs["data"]（pandas DataFrame），處理完回傳 {"data": result_df}，下游接著用：

# blocks/filter.py
async def execute(self, *, params, inputs, context) -> dict:
    df = inputs["data"]          # ← 上游傳來的 DataFrame
    column = params["column"]
    op = params["operator"]
    mask = _apply_op(df[column], op, params.get("value"))
    return {"data": df[mask]}    # → 傳給下游

為什麼用 DAG + pandas 而不是讓 LLM 寫 SQL？

  • LLM 生成 SQL 有 SQL injection 和誤刪資料的風險
  • DAG 每個 Block 的操作有限（filter/aggregate/chart），LLM 只填參數，不寫邏輯
  • 可以對 pipeline 做 dry-run / validate 而不擔心副作用
  • Block 有明確的 input_schema / output_schema，LLM 更容易正確生成
設計哲學：LLM 決定「要用哪些 block、參數填什麼」（語意）；DAG executor 決定「執行順序」（結構）；Block 的 Python 程式碼決定「實際運算」（邏輯）。三層各司其職。

    async def execute(self, *, params, inputs, context) -> dict:
        mcp_name = self.require(params, "mcp_name")  # 例如 "list_tools"
        args_raw = params.get("args") or {}

        # ① 向 Java 查 endpoint（每次呼叫都查一次）
        java = _make_java_client()
        mcp = await java.get_mcp_by_name(mcp_name)  # GET /internal/mcp-definitions
        # → {"name":"list_tools","api_config":"{\"endpoint_url\":\"http://...\",\"method\":\"GET\"}"}

        # ② 解析 api_config
        api_config = json.loads(mcp["api_config"])
        url    = api_config["endpoint_url"]
        method = api_config.get("method", "GET").upper()
        # method 白名單：只允許 GET / POST，否則拋 BlockExecutionError

        # ③ 發送請求
        async with httpx.AsyncClient(timeout=30.0) as client:

            else:
                resp = await client.post(url, json=args_raw)

        # ④ 回傳扁平化（自動偵測 JSON 結構）
        payload = resp.json()
        records = _flatten_response(payload)
        # 嘗試 key：events / dataset / items / data / records / rows
        # 都找不到就把整個 dict 當一列

        return {"data": pd.DataFrame(records)}

Q29
Sonnet tool-use loop 實際怎麼運作？

Sonnet 本身不執行任何 tool，它只輸出「我要呼叫哪個 tool + 什麼參數」的 JSON，code 去執行，結果再丟回給 Sonnet。

一次 LLM call 內部做了什麼

1. 組 system prompt（3 層）
   ├── Soul（工廠鐵律）          ← cache_control: ephemeral
   ├── MCP registry（工具清單）  ← cache_control: ephemeral
   └── RAG memories（動態記憶）  ← NOT cached

2. 決定哪些 tool 給 LLM 看
   ├── 隱藏內部工具（execute_mcp, query_data...）
   ├── ON_DUTY 角色 → 再隱藏寫入類（build_pipeline_live...）
   └── PIPELINE_ONLY_MODE → 隱藏 execute_skill

3. 呼叫 Anthropic SDK（streaming）
   max_tokens: 8192，最多 retry 2 次

4. 解析回應
   ├── 有 tool_use block → 交給 graph 執行 tool
   └── 純文字 → 路由到 synthesis，回應 user

Tool-use loop 流程

user 說「EQP-01 最近 OOC 情況」
    ↓
Sonnet → tool_use { get_process_info, {toolID: "EQP-01"} }
    ↓
code 打 HTTP 到 simulator（不需要 LLM）
    ↓
結果塞回 Sonnet context
    ↓
Sonnet 判斷：夠了 → 輸出純文字答案
    ↓
synthesis → 回應 user
終止條件（_should_continue routing function）：
LLM 回應無 tool_calls → synthesis（正常終止）
force_synthesis = true → synthesis（錯誤終止）
current_iteration ≥ 25 → synthesis（安全上限）

Q32
LLM 怎麼決定終止？會不會卡住不知道要停？

終止由 graph routing function 決定，不是 LLM 自己。LLM 決定 happy path；graph 保底「一定會停」。

Chat 端終止邏輯（_should_continue）

每次 llm_call 結束後 graph 判斷：

LLM 回應有 tool_calls？
  → YES → 執行 tool → 回 llm_call（繼續）
  → NO  → synthesis（終止，輸出給 user）

強制終止：
  → force_synthesis = True（錯誤）
  → current_iteration ≥ 25（安全上限）

Builder 端終止邏輯（phase loop）

每個 phase 的 ReAct round 結束後：

phase_done？
  → YES → 推進到下一個 phase 或 finalize
  → NO  → 繼續下一個 round

round 數 ≥ 8？
  → 強制 phase_revise 或 finalize

LLM 不知道要停的真實案例（Todos.md 記錄）

模式發生原因修法
re-inspect 迴圈tool 結果藏住 is_ooc 欄位，LLM 以為資料沒到，一直查inspect formatter 顯示全欄位
remove + re-add 同一 blockLLM 覺得沒成功，反覆拆掉重建prompt 明示此為浪費 round
false-successLLM 以為 phase 完成（其實沒有），下一 phase 又卡住ENABLE_STRICT_PHASE_OUTPUT
Brief 無窮迴圈bypass regex 不 match 空格 → 每輪重發 brief 卡regex 改 [^\]]*\]
設計原則：LLM 決定「繼續 or 停」的 happy path；graph 的 MAX_ITERATIONS / round budget 保底「最終一定停」。但上限只保證程式不死，不保證結果正確——這是 phase_revise 和 strict_phase_output 存在的原因。

Q36
Block 與 MCP 的關聯是什麼？怎麼呼叫？有 prompt 嗎？

大多數 block 與 MCP 無關。只有 SOURCE 的 4 個 block 有關聯，且呼叫方式各不相同。Prompt（LLM）只在 Agent 決定要填哪個 block 的參數時介入，block 實際執行是純 code，不再呼叫 LLM。

四種關係模式

① block_process_history — 完全 hardcode，繞過 mcp_definitions

對應 MCP：get_process_info（名義上），但實際上根本不查 DB。

# process_history.py
url = f"{ONTOLOGY_SIM_URL}/api/v1/process/info"   # 直接讀環境變數
resp = await httpx.get(url, params={toolID, lotID, step, ...})
改 mcp_definitions 裡 get_process_info 的 api_config 完全不影響此 block 的行為。

② block_rework_request — 1:1 專屬 MCP，有自己的 executor

對應 MCP：rework_request（V53 同時 INSERT block + MCP）。

# rework_request block executor（自有邏輯）
POST http://localhost:8012/api/v1/rework_request
body: { lotID, step?, flowID? }
→ 回傳後展平 reworkInfo.* 欄位成 rwi_* 前綴 columns
雖有對應 MCP，但 block 不走 block_mcp_call 通用 dispatcher，有自己的 executor 來處理 reworkInfo 的欄位展平邏輯。

③ block_list_objects — KIND_TO_MCP 對照表 → 委派給 block_mcp_call

對應 MCP：5 個 list 型 MCP（tool/lot/step/apc/spc）。

# list_objects.py
KIND_TO_MCP = {
    "tool": "list_tools",
    "lot":  "list_active_lots",
    "step": "list_steps",
    "apc":  "list_apcs",
    "spc":  "list_spcs",
}
# 實際呼叫：委派給 McpCallBlockExecutor
mcp_name = KIND_TO_MCP[kind]
return await self._mcp.execute(params={"mcp_name": mcp_name, "args": args}, ...)

設計目的：讓 Agent 只需填 kind="tool"，不需要記 MCP 名稱（提升 discoverability）。底層走一樣的 dispatch path。

④ block_mcp_call / block_mcp_foreach — 通用動態路由

對應 MCP：任何已在 mcp_definitions 登記的 MCP（最常用：get_process_summary、get_process_info 等）。

Q42
Verifier 結構 vs 語意邊界：它檢查什麼、不檢查什麼

Verifier（phase_verifier.py）是 純結構性保證，不是語意正確性保證。v30.20 明確移除了 rows 品質閘門與 LLM-judge，改為「runtime concern」。

檢查項目誰負責方式
block executor 是否拋錯（Check B）✅ Verifier讀 error / validation_error 欄位
孤立節點（無入邊，Check C）✅ Verifiergraph traversal
葉節點（非 output 無出邊，Check C2）✅ Verifiergraph traversal；3 次後強制 prune
phase kind 覆蓋（Check A，STRICT mode）✅ Verifierterminal block 的 covers 集合對比 expected
chart 最終渲染 + vision judge（F2）✅ Verifier（Haiku）Playwright 截圖 → Haiku 看「有沒有空圖」
chart encoding gap（P2a）✅ Verifierphase 說「顏色區分」但 series_field 未設 → REJECT
欄位名稱是否正確❌ 無人在 build 時檢查—
filter 條件語意對不對❌ 無人在 build 時檢查—
資料行數是否合理❌ v30.20 已移除原本有，現已刪

語意錯誤的實際後果

若 Builder 把欄位名填錯（e.g. 應該是 temperature 但寫成 temp）：

Builder 建 block → Verifier 全部 PASS（結構正確）
→ Pipeline 執行：transform block 過濾出空 DataFrame
→ Chart 顯示空圖 or 全 NaN
→ 這時 F2 vision judge 才有機會抓到（空圖 → REJECT）
→ 但若「數值有出來只是錯的」→ vision judge 也抓不到 → user 直接看到錯誤結果
設計理由：欄位名稱驗證需要真的去打 ontology 拿資料才能確認（等於執行 pipeline），那就不是 "verify" 而是 "execute"。Verifier 的價值正是「純 code、快速、不用 LLM」，加語意驗證會讓整個快速回饋迴圈失效。Vision judge 是唯一的語意層 safety net，但只抓視覺異常，不抓數值錯誤。

① Planner — goal_plan.py _SYSTEM（最長，約 200 行）

⑤ Repairer — phase_revise.py _SYSTEM（最短）

你會收到：
  - CURRENT PHASE goal + expected
  - 完整 action history（你試過什麼 tool + 結果）
  - 當前 canvas state

  2. 提出 1 個明確的 alternative 策略（e.g. 「改用 X block 而非 Y」）

  "alternative_strategy": "...",

Q45
Builder ReAct Loop：每個行動都是一次獨立 LLM call 嗎？

不是每個行動各自一次 LLM call。正確理解是：整個 round 是一次 LLM call，LLM 在那次 call 裡決定要做哪一個行動，然後等系統執行結果，下一個 round 再做下一次 LLM call。

一個 Phase 的完整流程（範例）

round 1：
  → 系統把 [phase 目標 + canvas 狀態 + 可用 tools] 送給 LLM（1 次 LLM call）
  ← LLM 輸出 tool_use：inspect_node_output(node_id="n1")
  → 系統執行 inspect，拿到 upstream schema（純 code，不用 LLM）
  → 把結果塞進下一輪 context

round 2：
  → 系統把 [phase 目標 + canvas 狀態 + round 1 結果] 送給 LLM（1 次 LLM call）
  ← LLM 輸出 tool_use：add_node(block_id="block_filter", params={...})
  → 系統執行 add_node + preview snapshot 更新（純 code）

round 3：
  → LLM 看到 add_node 結果（1 次 LLM call）
  ← LLM 輸出 tool_use：run_verifier()
  → 系統跑 Verifier（純 code），回 ACCEPT 或 REJECT

ACCEPT → 推進下一 phase
REJECT → round 4：LLM 看到 reject 原因，決定 set_param 或換 block

數字整理

概念說明
1 round = 1 次 LLM callLLM 每次只決定「下一步做什麼」，輸出一個 tool_use JSON
1 phase = 多個 round通常 2–5 round（inspect → add_node → verify），上限 8 round（chart/alarm 12）
Tool 執行本身不用 LLMinspect / add_node / connect / set_param / run_verifier 都是系統執行的 code
「行動順序」是建議prompt 裡列的順序是給 LLM 看的參考，LLM 每次 call 自己選哪個行動

對比：整個 Pipeline build 的 LLM call 次數

Planner：            1 次（Haiku，一次出所有 phases）
Builder（per phase）：2–5 次（Haiku，每 round 1 次）
Verifier：           0 次（純 code）+ 偶爾 1 次 vision judge（Haiku）
Repairer（發生時）：  1–2 次（Haiku）

順利建一個 3-phase pipeline：
  最少：1 + 3×2 = 7 次 Haiku call
  常見：1 + 3×3 = 10 次 Haiku call
  失敗修復：可達 15–20 次
設計原則：每個 round 只發一個 tool call（prompt 明確規定），是為了讓系統在每次行動後都能插入 preview snapshot 與 verifier 資訊，確保 LLM 每輪都能看到最新的 canvas 狀態。如果允許一次 emit 多個 action，中間狀態會混亂，debug 也更難。
